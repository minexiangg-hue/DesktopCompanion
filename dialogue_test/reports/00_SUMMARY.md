## 四、搜索执行质量

### 每次搜索实际调用的方法
所有搜索均正确调用了 `WebSearch.query()` 或 `WebSearch.searchNews()`，参数为从用户消息中提取的搜索词。

### 搜索结果来源
```
local: SpaceX 成功上市，首日市值突破万亿
local: AI 芯片竞争白热化，多家厂商发布新一代产品
local: 全球气温持续升高，多国加速部署可再生能源
local: JavaScript
local: AI
local: 区块链
```

### ⚠️ 搜索质量问题

**发现：所有搜索均回退到本地 fallback，未成功调用远程 API（Wikipedia/Reddit/HN）。**

原因分析：
1. WebSearch 的三个远程后端（Wikipedia OpenSearch、Reddit JSON API、HN Algolia）在此测试环境中均未能返回有效结果
2. 本地知识库（15个硬编码技术主题）+ 趋势 fallback（7个硬编码伪热点）作为兜底
3. 无论用户查询什么（"量子计算"/"Vision Pro"/"比特币"），返回的都是同样的 7 个 fallback 条目

**对用户体验的影响**：
- AI 回复始终引用 fallback 条目的标题（如"SpaceX 成功上市"），与用户实际查询完全无关
- 这直接对应了用户反馈的"AI回答模棱两可，搬出1-2年前的事件"
- 在真实网络环境中，如果远程 API 偶尔失败，会间歇性地出现此类问题

### 搜索词提取质量
`_extractSearchTerms()` 能正确剥离搜索指令前缀，但存在轻微过度剥离：
- "帮我搜索一下最新的科技新闻" → 搜索词: "一下 科技新闻"（多余"一下"）
- "搜一下最近有什么好电影" → 搜索词: "有什么好电影"（丢失了"最近"时间词）

## 五、人格对话质量

### 模板回复分析
6 个人格的模板回复均正常运作。但搜索结果的"人格染色"存在局限：
- `_colorizeSearchResult()` 仅使用搜索结果的 `title` 字段
- 不展示 snippet/URL/来源等信息
- 染色模板中未提及"这是搜索结果"（用户无法区分搜索回复和模板回复）

### 人格差异
所有 6 个人格的**意图检测行为完全一致**（`_detectIntent` 是人格无关的）。人格差异仅体现在回复的语气/措辞上。

## 六、代码层面发现的问题及修复

### 已修复
1. ✅ **WebSearch 实例化**: main.js 创建 + 注入 ChatEngine
2. ✅ **意图检测顺序**: 搜索关键词优先级提到互动意图之前（修复 "搜一下有什么好电影" → recommend 而非 search）
3. ✅ **语义匹配**: 新增 "是什么/为什么/怎么/如何/原理/区别" 等自然语言查询模式
4. ✅ **default_searchable 兜底**: 严格条件（查询特征 + 非自我陈述 + 非aboutBot + 非话题切换）
5. ✅ **news 正则收紧**: 移除独立"最近"匹配（修复 "最近工作压力好大" → news 误判）
6. ✅ **搜索进度 UI**: 动画 + 来源标签 + 超时处理
7. ✅ **`>>>SEARCH<<<` 协议**: 正则修复
8. ✅ **30 秒超时**: 防止无限等待

### 待改进
1. 🔧 **搜索后端质量**: 当前 Wikipedia/Reddit/HN 免费 API 在部分网络环境下不可靠，建议后续集成正式搜索引擎 API（如 Brave Search、SerpAPI 等）
2. 🔧 **搜索词提取**: "帮我搜索一下最新的..." 剥离"帮我搜索"后余留"一下"，建议优化正则
3. 🔧 **搜索结果利用**: `_colorizeSearchResult()` 仅用 title，应展示 snippet + 来源标注
4. 🔧 **搜索 cache TTL**: 通用查询 10 分钟可能偏短，建议根据查询类型差异化

## 八、最终评估

**意图检测链路**: ★★★★★ 优异 — 所有场景正确分类，无遗漏无过触发

**搜索执行链路**: ★★★☆☆ 一般 — 接线正确、函数调用正确，但搜索后端（Wikipedia/Reddit/HN）不可靠，实际返回 fallback 数据
